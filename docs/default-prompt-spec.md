# Default Prompt Spec

This document proposes stronger default prompts for `ralph++` review and fix
loops. The goal is to make prompt behavior more explicit, reduce ambiguous
output, and improve the stability of reviewer/fixer handoff.

## Prompt Table

| Prompt | Used By | Placeholders | Output Contract | Proposed Default |
| --- | --- | --- | --- | --- |
| `prd_review.reviewer_prompt` | PRD review loop | `{prd_file}` | Output exactly `LGTM` if the PRD is acceptable. Otherwise output a numbered list of issues. Each issue should include `severity`, `section`, `problem`, `consequence`, and `recommended fix`. | ```text\nRead the PRD at {prd_file}.\n\nEvaluate whether it is:\n- complete\n- unambiguous\n- implementable as a sequence of small independent user stories\n- testable with concrete acceptance criteria\n- explicit about constraints, edge cases, and non-goals where relevant\n\nIf the PRD is fully satisfactory, output exactly:\nLGTM\n\nOtherwise, output a numbered list of issues.\n\nFor each issue include:\n- severity: critical | major | minor\n- section: the relevant PRD section or story\n- problem: what is unclear, missing, inconsistent, or too broad\n- consequence: why this would cause implementation or review problems\n- recommended fix: the smallest concrete change that would resolve it\n\nDo not rewrite the PRD yourself. Only review it.\n``` |
| `prd_review.fixer_prompt` | PRD review loop | `{prd_file}`, `{findings}` | Edit the PRD in place. Do not replace edits with a summary. | ```text\nThe following issues were found in the PRD at {prd_file}:\n\n{findings}\n\nRevise the PRD in place to resolve these issues.\n\nRequirements:\n- preserve the original feature intent unless the findings require clarification\n- keep stories small, implementation-ready, and independently testable\n- add or tighten acceptance criteria where needed\n- make ambiguities explicit rather than leaving them implied\n- keep the document concise and structured\n- do not add speculative scope that is not justified by the feature request or findings\n\nDo not output a summary instead of making the edits. Update the PRD itself.\n``` |
| `post_review.reviewer_prompt` | Post-run review loop | `{prd_file}` | Output exactly `LGTM` if the implementation is acceptable. Otherwise output a numbered list of findings. Each finding should include `severity`, `file`, `problem`, `evidence`, and `recommended fix`. | ```text\nRead the implementation and the requirements in {prd_file}.\n\nReview whether the code fully satisfies every required story and acceptance criterion.\n\nCheck for:\n- missing functionality\n- incorrect behavior\n- regressions\n- edge-case handling\n- obvious design or contract violations\n- missing or inadequate tests\n- mismatches between the implementation and the PRD\n\nIf everything looks correct, output exactly:\nLGTM\n\nOtherwise, output a numbered list of findings.\n\nFor each finding include:\n- severity: critical | major | minor\n- file: exact path(s) if applicable\n- problem: what is wrong or missing\n- evidence: the concrete mismatch, bug risk, or regression\n- recommended fix: the smallest reasonable corrective action\n\nBe specific. Do not give vague style feedback unless it affects correctness or maintainability materially.\n``` |
| `post_review.fixer_prompt` | Post-run review loop | `{prd_file}`, `{findings}` | Fix the implementation in place. Preserve correct changes and avoid unrelated edits. | ```text\nThe following issues were found in the final implementation against {prd_file}:\n\n{findings}\n\nFix these issues in the codebase.\n\nRequirements:\n- address every finding unless two findings are duplicates\n- preserve already-correct behavior\n- avoid unrelated refactors\n- add or update tests where needed\n- keep changes minimal but sufficient\n- ensure the implementation remains aligned with the PRD\n\nDo not just describe the fixes. Make the code changes.\n``` |
| `orchestrated.review_prompt` | Iteration review inside orchestrated mode | `{prd_file}`, `{diff}` | Output exactly `LGTM` if the latest iteration is acceptable. Otherwise output a numbered list of findings. Each finding should include `severity`, `file`, `problem`, `evidence`, and `recommended fix`. | ```text\nReview the latest iteration against the requirements in {prd_file}.\n\nStart from this git diff:\n\n{diff}\n\nYou may inspect the changed files and nearby code as needed.\n\nCheck for:\n- requirement mismatches\n- broken or incomplete behavior\n- regressions\n- missing edge-case handling\n- unsafe assumptions\n- missing tests or inadequate test updates\n\nIf the iteration is acceptable, output exactly:\nLGTM\n\nOtherwise, output a numbered list of findings.\n\nFor each finding include:\n- severity: critical | major | minor\n- file: exact path(s) if applicable\n- problem: what is wrong, risky, or incomplete\n- evidence: what in the diff or code supports the finding\n- recommended fix: the smallest reasonable corrective action\n\nOnly report findings that materially affect correctness, completeness, or reliability.\n``` |
| `orchestrated.fix_prompt` | Iteration fix step inside orchestrated mode | `{prd_file}`, `{findings}` | Fix the repository in place. Keep the patch targeted and preserve correct existing work. | ```text\nThe following issues were found in the latest code changes against {prd_file}:\n\n{findings}\n\nFix these issues in the repository.\n\nRequirements:\n- resolve each finding concretely\n- preserve correct existing changes\n- avoid unrelated edits\n- keep the patch as small as possible while fully fixing the problems\n- update tests if needed\n- do not claim success without making the changes\n\nIf some finding is invalid or already resolved, handle that conservatively and focus on the remaining real issues.\n``` |
| fallback coder prompt | Host-side fallback written to `scripts/ralph/CLAUDE.md` in orchestrated mode when no repo-local `CLAUDE.md` exists | none | Implement only the next smallest incomplete story. Emit `<promise>COMPLETE</promise>` only when all stories are done. | ```text\nRead scripts/ralph/prd.json and implement the next smallest incomplete user story.\n\nRequirements:\n- make only the changes needed for that story and its acceptance criteria\n- inspect the surrounding code before editing\n- preserve existing behavior unless the story requires a change\n- update or add tests when appropriate\n- keep the repository in a coherent, runnable state\n- commit your changes if your workflow expects it\n\nIf all stories are complete, output exactly:\n<promise>COMPLETE</promise>\n\nDo not output that completion signal unless every story in prd.json is complete.\n``` |

## Design Notes

| Principle | Rationale |
| --- | --- |
| Strict `LGTM` terminal state | Keeps reviewer output machine-checkable and avoids accidental false positives. |
| Structured findings | Makes fixer prompts more actionable and reduces vague prose. |
| Minimal corrective action | Reduces iteration thrash and unrelated edits. |
| Explicit evidence requirement | Pushes reviewers toward concrete, code-review-quality findings. |
| Stronger fixer instructions | Encourages actual repository edits instead of narrative responses. |
| Conservative completion signal | Reduces false completion in orchestrated mode. |
